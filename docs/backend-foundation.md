# Backend Foundation Architecture

## Overview
This document outlines the architectural decisions for the Express.js and TypeScript backend foundation, ensuring it is production-ready for deployment to Render and connecting to Neon PostgreSQL.

## Core Libraries
- **Express**: HTTP server framework.
- **Zod**: Runtime type validation (used for strict Environment Variable checking).
- **Pino & Pino-HTTP**: High-performance structured logging.
- **Helmet**: Security HTTP headers.
- **CORS**: Cross-origin resource sharing, strictly bound to the frontend URL.
- **express-async-errors**: Patches Express to automatically catch async route errors without requiring repetitive `try/catch` blocks.
- **Prisma**: Type-safe ORM for PostgreSQL.

## Startup Lifecycle
The application strictly separates configuration from execution to ensure testability and reliable process management.

1. `src/config/env.ts`: Loads `.env` and enforces validation via Zod. Fails the process immediately on invalid configuration.
2. `src/app.ts`: Instantiates the Express application and attaches middlewares, routes, and global error handlers.
3. `src/server.ts`: Imports the configured Express app, starts the HTTP listener on the configured port, and attaches OS signal listeners for graceful shutdown.

## Middleware Order
Global middlewares are registered in `src/app.ts` in the following strict order:
1. `helmet()` - Security headers.
2. `cors()` - Cross-Origin Request parsing.
3. `compression()` - Response payload compression.
4. `express.json()` / `express.urlencoded()` - Body parsing (limited to 1MB).
5. `pinoHttp()` - Structured HTTP request logging.
6. **API Routers** - e.g., `/api/v1`
7. **Not Found Handler** - Catches unresolved routes and throws a `NotFoundError`.
8. **Global Error Handler** - Intercepts all thrown errors.

## Prisma Lifecycle
- **Instantiation**: A single Prisma client instance is managed at `src/db/prisma.ts`.
- **Query Logging**: Prisma query/warn/error logs are piped directly into the central Pino logger.
- **Shutdown**: The process termination gracefully invokes `prisma.$disconnect()`.

## Error Architecture
Errors are strictly categorized using the `AppError` base class:
- **Operational Errors** (e.g., `ValidationError`, `NotFoundError`): Known errors generated by business logic or bad requests. Handled gracefully, returning structured 4xx JSON.
- **Unexpected Errors**: Code crashes or 500s. Handled by logging the stack trace (in development) and returning a generic 500 JSON payload to the user without leaking internal state.

## Logging Strategy
- **Development**: Pino outputs formatted, colorized text using `pino-pretty`.
- **Production**: Pino outputs raw, structured JSON for seamless ingestion by logging aggregators (e.g., Datadog, AWS CloudWatch).
- **Redaction**: Request logging ignores the high-frequency `/api/v1/health` endpoint to prevent log noise. Sensitive data is kept out of logs by default design.

## Graceful Shutdown
To satisfy Render deployment constraints, the backend catches `SIGINT` and `SIGTERM`.
1. Logs the shutdown intent.
2. Stops the HTTP server (refusing new connections).
3. Disconnects the Prisma database connection cleanly.
4. Exits with process code 0.
Uncaught exceptions and unhandled rejections trigger the same shutdown lifecycle, ensuring the process does not linger in an unstable state.
